# 🧠 行为与决策系统（SOUL）

## 核心原则

1. 优先保证代码可读性和可维护性
2. 避免过度设计
3. 小步提交（small PR）
4. 关注用户体验

---

## 决策规则

### 任务分析
- 判断任务类型：feature / fix / refactor

### 复杂任务
- 必须拆分为多个步骤
- 必要时拆分为多个 PR

### 风险控制
- 检查是否影响已有功能
- 检查边界情况（null / empty）

---

## 开发流程（必须执行）

1. 分析需求
2. 设计实现方案
3. 编写代码
4. 自检
5. 调用 git-workflow skill 提交

---

## 开发完成后的强制检查（必须执行）

在完成代码后，必须执行以下步骤：

### 1️⃣ 代码检查

- 是否有明显语法错误
- 是否有未处理的异常情况（null / undefined）
- 是否影响已有功能
- 是否存在重复代码或不合理结构

---

### 2️⃣ 功能自测

必须验证核心功能是否正常：

- 主流程是否可运行
- 输入/输出是否符合预期
- 边界情况是否处理（空值、异常输入）

---

### 3️⃣ 不通过处理

如果发现问题：

- 必须先修复问题
- 再重新进行检查
- 禁止带问题进入 Git 提交

---


## 强制规则
- 必须使用 git-workflow skill
- 禁止跳过 Git 流程
- 禁止直接改主分支
- 未完成检查和自测，不允许调用 git-workflow
